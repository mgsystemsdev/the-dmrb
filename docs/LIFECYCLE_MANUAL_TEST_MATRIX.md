# Lifecycle Manual Test Matrix

Manual test cases for lifecycle entry, board visibility, report order, repair workflows, and invariants.  
For each test: **Setup**, **Action**, **Expected result**, **Files/components involved**.

| # | Test scenario | Setup | Action | Expected result | Files / components involved |
|---|----------------|-------|--------|-----------------|-----------------------------|
| 1 | **Move-out arrives first** | Unit exists in unit master; no open turnover. Property has task templates. | Import MOVE_OUTS CSV with unit + move-out date. | One open turnover created with `move_out_date` set; tasks instantiated; turnover appears on board; SLA/risk reconciled. | `services/imports/move_outs.py`, `services/imports/tasks.py`, `db/repository/turnovers.py`, `services/turnover_service.py`, `services/board_query_service.py` |
| 2 | **Move-in arrives first** | Unit exists; no open turnover. | Import PENDING_MOVE_INS CSV with unit + move-in date. | No turnover created. `import_row` has `validation_status=CONFLICT`, `conflict_reason=MOVE_IN_WITHOUT_OPEN_TURNOVER`. Row appears in Missing Move-Out queue. | `services/imports/move_ins.py`, `db/repository/imports.py`, `services/report_operations_service.py`, `ui/screens/report_operations.py` |
| 3 | **Availability arrives first** | Unit exists; no open turnover. | Import AVAILABLE_UNITS or DMRB with unit + ready date. | No turnover created. `import_row` has IGNORED / `NO_OPEN_TURNOVER_FOR_READY_DATE`. Unit does not appear on board. | `services/imports/available_units.py`, `services/imports/dmrb.py`, `db/repository/turnovers.py` |
| 4 | **Move-in without move-out** | Same as #2: unit has move-in in report but no move-out. | Open Report Operations → Missing Move-Out tab. | Unit listed with conflict reason (MOVE_IN_WITHOUT_OPEN_TURNOVER or MOVE_OUT_DATE_MISSING). Resolve section shows "Select unit to resolve" and move-out date input. | `services/report_operations_service.py` (`get_missing_move_out_queue`), `db/repository/imports.py`, `ui/screens/report_operations.py` |
| 5 | **Missing move-out resolved** | Unit in Missing Move-Out queue (no open turnover). | In Report Operations → Missing Move-Out: select unit, enter move-out date, click "Create turnover". | One open turnover created with given move-out date; tasks instantiated; unit disappears from Missing Move-Out queue and appears on board. | `ui/screens/report_operations.py`, `services/report_operations_service.py` (`resolve_missing_move_out`), `services/manual_availability_service.py`, `services/turnover_service.py`, `services/imports/tasks.py` |
| 6 | **FAS note persistence** | Unit has open turnover or at least PENDING_FAS import row for property. | Report Operations → FAS Tracker: add or edit note for a unit/fas_date. Save. Re-import PENDING_FAS (or any later import). Open FAS Tracker again. | Note still present for that unit/fas_date. Notes persist across imports. | `ui/screens/report_operations.py`, `services/report_operations_service.py` (`get_fas_tracker_rows`, `upsert_fas_note`), `db/repository/fas_tracker_notes.py`, `db/repository/imports.py` |
| 7 | **Import diagnostics deduplication** | Multiple import batches with non-OK rows for same (unit_code_norm, report_type). | Open Report Operations → Import Diagnostics tab. | One row per (unit_code_norm, report_type)—most recent by `imported_at`. No duplicate rows for same unit+report type. | `db/repository/imports.py` (`get_import_diagnostics`), `services/report_operations_service.py` (`get_import_diagnostics_queue`), `ui/screens/report_operations.py` |
| 8 | **Duplicate turnover prevention** | Unit already has one open turnover. | Try to create a second open turnover: (a) import MOVE_OUTS for same unit again with same move-out, or (b) Report Operations resolve for same unit, or (c) Manual Availability for same unit. | (a) MOVE_OUTS updates existing turnover or CONFLICT; no second turnover. (b) Resolve only lists units with no open turnover—if unit already resolved, not in queue. (c) Manual Availability raises ValueError: "This unit already has an open turnover." | `services/imports/move_outs.py`, `services/manual_availability_service.py`, `db/repository/turnovers.py` (`get_open_turnover_by_unit`), `services/report_operations_service.py` |
| 9 | **Board shows only active turnovers** | DB has open turnover rows: some with `move_out_date` set, some with `move_out_date` NULL (pre-turn). | Open DMRB Board (or any view using list_open_turnovers / list_open_turnovers_by_property). | Only turnovers with `move_out_date IS NOT NULL` appear. Pre-turn or placeholder rows (move_out_date NULL) never appear on board. | `db/repository/turnovers.py` (`list_open_turnovers`, `list_open_turnovers_by_property`), `services/board_query_service.py`, `ui/screens/board.py` |
| 10 | **Lifecycle phase derivation** | Open turnover with move_out_date, optionally move_in_date; today known. | View board or turnover detail; phase/NVM label shown (e.g. VACANT, SMI, MOVE_IN_COMPLETE, NOTICE). | Phase matches spec: e.g. today ≥ move_out and no move_in → VACANT; today in [move_in, move_in+14] → MOVE_IN_COMPLETE or STABILIZATION; move_out NULL → NOTICE/NOTICE_SMI. NVM (N/V/M) derived from phase. | `domain/lifecycle.py` (`derive_lifecycle_phase`, `effective_move_out_date`), `domain/enrichment.py` (`derive_phase`, `compute_facts`), `services/board_query_service.py`, `ui/screens/board.py`, `ui/screens/turnover_detail.py` |
| 11 | **Readiness independent from tasks** | Open turnover with move_out_date; tasks may be incomplete or not started. | Set Manual Ready status (e.g. Ready / Not Ready) and/or confirm manual ready on turnover detail. | Manual ready status and manual_ready_confirmed_at update without requiring tasks to be complete. SLA/risk reconcile run; readiness and task state are independent. | `services/turnover_service.py` (`set_manual_ready_status`, `confirm_manual_ready`), `domain/sla_engine.py`, `services/sla_service.py`, `ui/screens/turnover_detail.py` |

---

## Reference

- **Lifecycle spec:** `docs/LIFECYCLE_STATE_AND_WORKFLOW_SPECIFICATION.md`
- **Board data:** `services/board_query_service.py` → `repository.list_open_turnovers` / `list_open_turnovers_by_property`
- **Repair flow:** Report Operations → Missing Move-Out → resolve = create turnover with move-out via `manual_availability_service.add_manual_availability`
- **Import order:** Reports can arrive in any order; only MOVE_OUTS creates turnovers; move-in/availability without move-out produce exceptions and appear in Report Operations.
